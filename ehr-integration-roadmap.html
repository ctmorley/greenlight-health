SECTION 02

SMART on FHIR: The Universal Foundation

SMART (Substitutable Medical Applications, Reusable Technologies) on FHIR is the ONC-mandated standard for launching third-party apps inside any compliant EHR. Current version: v2.2.0.

Two Launch Patterns

EHR Launch (Primary)

App launches from within an existing EHR session. The EHR passes an opaque launch context handle; the app inherits the user's session context (current patient, encounter, practitioner). This is the primary integration mode for clinical workflows.

Point-of-order Context-aware

Standalone Launch

App launches independently from a browser or mobile device, then authenticates against the EHR's FHIR server. Used for administrative workflows, batch processing, and non-clinical users.

Admin workflows Batch processing

OAuth 2.0 + PKCE Authentication Flow

// 1. Discovery GET {fhir_base}/.well-known/smart-configuration // Returns: authorization_endpoint, token_endpoint, scopes_supported // 2. Authorization Request GET {authorization_endpoint}? response_type=code& client_id={app_id}& redirect_uri={callback}& scope=launch patient/Patient.read patient/Coverage.read& state={random}& code_challenge={pkce_challenge}& code_challenge_method=S256 // 3. Token Exchange (after user authorizes) POST {token_endpoint} grant_type=authorization_code& code={auth_code}& code_verifier={pkce_verifier} // 4. Returns: access_token, refresh_token, patient (context ID)

SECTION 03

Da Vinci Implementation Guides: CRD, DTR, PAS

The HL7 Da Vinci Project defines the three-part electronic Prior Authorization (ePA) framework. These are the standards that CMS-0057-F recommends payers adopt by January 2027.
The integrated workflow: CRD detects PA requirement at point of order → DTR auto-collects documentation from EHR → PAS submits the complete PA request electronically → Payer responds with approval/denial in real-time. This eliminates fax, phone, and portal submissions entirely.

CRD — Coverage Requirements Discovery

Real-time "is PA required?" check at the point of order. Uses CDS Hooks to intercept clinical workflow events inside the EHR.

Mandatory hooks: appointment-book, orders-sign, order-dispatch

Returns: Coverage status, PA requirement (no-auth / auth-needed / satisfied), documentation requirements, payer-specific rules

GreenLight advantage: Our existing ACR criteria engine (11,975 guidelines) and payer policy database power the CRD response. This intelligence is already built — CRD is the delivery mechanism.

CDS Hooks Real-time v2.2.1 STU

DTR — Documentation Templates & Rules

Auto-populates documentation requirements by pulling clinical data directly from the EHR. Payers publish FHIR Questionnaire resources with embedded CQL logic.

Workflow: SMART on FHIR app presents Questionnaire → CQL engine auto-fills answers from EHR data → clinician confirms → QuestionnaireResponse generated

Data auto-populated: Clinical notes, lab values, prior imaging results, medication history, diagnosis history, treatment records

SMART launch CQL engine v2.2.0 STU

PAS — Prior Authorization Support

Electronic submission of PA requests from EHR to payer via the Claim/$submit FHIR operation. Replaces fax, phone, and portal submissions entirely.

Submission bundle: Claim + Patient + Coverage + ServiceRequest + Practitioner + Organization + Condition + Observation + QuestionnaireResponse

Response types:

  • Approved — with authorization number
  • Denied — with specific reason code
  • Pended — with request for additional info

X12 compatibility: Supports conversion to/from X12 278 for backward compatibility with payers not yet on FHIR.

$submit Real-time v2.1.0 STU
Why this matters for GreenLight: Today, our PA wizard requires 5 steps of manual data entry. With CRD + DTR + PAS, the workflow becomes: clinician orders an MRI → CRD confirms PA is needed → DTR auto-fills 90%+ of the documentation → PAS submits instantly. The clinician's only action is placing the order. Everything else is automated.

SECTION 06

Epic Integration: Step-by-Step

Epic has ~38% of US hospital market share and is the highest-priority EHR integration target. Here is the complete process from zero to production.

Step 1: Developer Registration

  • Register at open.epic.com (free, immediate access)
  • Access FHIR documentation, sandboxes, and learning resources
  • Review Epic's 450+ FHIR R4 API endpoints across 55 resource types
Timeline: 1–2 days

Step 2: App Registration & Scoping

  • Define exact FHIR resources, scopes, and launch context
  • Choose EHR Launch (primary) + Standalone Launch (admin)
  • Register redirect URIs and configure PKCE
Timeline: 1 week

Step 3: Sandbox Development

  • Build SMART on FHIR launch flow against Epic public sandbox
  • Implement OAuth 2.0 + PKCE token exchange
  • Test FHIR resource reads: Patient, Coverage, Condition, ServiceRequest
  • Build auto-fill mapping to GreenLight wizard steps
  • Validate against synthetic patient data
Timeline: 2–4 months

Step 4: Customer Sponsorship

  • Find at least one Epic customer site willing to sponsor the app
  • Customer submits activation request on GreenLight's behalf
  • Target: early GreenLight imaging center customer running Epic
Timeline: 1–3 months (critical gate)

Step 5: Epic Technical Review

  • Epic engineers review FHIR compliance and SMART implementation
  • Functionality, performance, and UX review
  • Must demonstrate proper scope handling and error cases
Timeline: 2–8 weeks

Step 6: Security Validation

  • Penetration testing and vulnerability assessment
  • HIPAA compliance verification
  • TLS 1.2+ transport encryption
  • OAuth 2.0 with PKCE (mandatory)
  • AES-128+ encryption at rest
  • SOC 2 Type II report preferred (not always mandatory at this stage)
Timeline: 2–4 weeks

Step 7: Connection Hub Listing

  • Execute agreements with Epic Vendor Services
  • Annual listing fee: ~$500/year
  • Listed in Showroom → visible to all Epic customers
Timeline: 1–2 weeks

Step 8: Per-Site Activation

  • Each Epic customer individually activates GreenLight
  • Site-specific configuration and data mapping
  • User acceptance testing at each site
  • Go-live with monitoring and support
Timeline: 2–4 weeks per site
The critical gate is Step 4: Customer Sponsorship. You cannot get an Epic Showroom listing without a live customer. Start building relationships with Epic-running imaging centers now, even before the SMART on FHIR code is complete. The sponsorship process often takes longer than the technical development.

Implementation Phases & Timeline

A 6-phase roadmap from current state to multi-EHR, fully automated PA submission. Phases overlap to maximize velocity.

Phase 0: Foundation (Weeks 1–4)

  • Add FHIR R4 libraries (fhirclient, @smile-cdr/fhirts)
  • Create SMART on FHIR launch endpoint (/launch)
  • Implement OAuth 2.0 + PKCE flow
  • Set up FHIR data extraction service layer
  • Begin SOC 2 Type II preparation (parallel)
Milestone: SMART launch working in sandbox

Phase 1: FHIR Data Auto-Fill (Weeks 2–8)

  • Map FHIR Patient → Step 1 patient fields
  • Map FHIR Coverage → Step 3 insurance fields
  • Map FHIR Condition → Step 2 ICD-10 codes
  • Map FHIR ServiceRequest → Step 2 CPT/procedure
  • Map FHIR Practitioner → Step 3 physician fields
Milestone: 80%+ of wizard auto-filled from EHR

Phase 2: Da Vinci CRD (Weeks 6–12)

  • Build CDS Hooks service endpoint
  • Implement order-sign, appointment-book hooks
  • Connect to existing ACR criteria engine
  • Return PA requirement + documentation needs
Milestone: Real-time "is PA required?" in EHR

Phase 3: Da Vinci DTR (Weeks 10–16)

  • Build FHIR Questionnaire renderer
  • Integrate CQL execution engine for auto-population
  • Pull clinical docs via DocumentReference
  • Generate QuestionnaireResponse for submission
Milestone: Auto-populated documentation package

Phase 4: Da Vinci PAS (Weeks 14–20)

  • Build FHIR Bundle assembly (Claim + supporting resources)
  • Implement Claim/$submit operation
  • Parse ClaimResponse for approval/denial/pend
  • Map responses to GreenLight status tracking
  • Add X12 278 fallback via Mirth Connect/Redox
Milestone: End-to-end electronic PA submission

Phase 5: Epic Production (Weeks 8–24)

  • Secure customer sponsor for Epic activation
  • Complete Epic technical + security review
  • Connection Hub listing
  • First production go-live at pilot site
Milestone: Live in Epic production environment

Critical infrastructure note: Vercel's standard Hobby/Pro plans do not include a BAA. HIPAA-compliant hosting requires either Vercel Enterprise or migrating to a BAA-covered platform (AWS with BAA, Azure with BAA). This must be resolved before any real patient data enters the system. Neon also requires confirming HIPAA eligibility on the current plan tier.
HHS provides model BAA language at hhs.gov/hipaa. Key requirement: BAAs must be executed before any PHI is disclosed to the business associate. Review annually and whenever scope of services changes. All BAAs should include breach notification obligations (within 60 days per HIPAA), subcontractor flow-down requirements, and return/destruction of PHI upon termination.
